# Отчетное описание проекта Cinema Info System

## 1. Назначение проекта

Cinema Info System - это веб-приложение для управления информацией о кинотеатрах, фильмах, залах, сеансах, билетах, пользователях и промокодах.

Система позволяет обычным пользователям просматривать кинотеатры, фильмы и сеансы, регистрироваться, входить в аккаунт, пополнять баланс через промокоды и покупать билеты на конкретные места. Администратор может управлять справочниками, пользователями, промокодами, импортировать и экспортировать данные.

Основная идея проекта - создать единую информационную систему кинотеатров с разделением прав доступа и сохранением истории купленных билетов.

## 4. Архитектура приложения

Проект построен по клиент-серверной архитектуре.

Frontend работает в браузере и отвечает за:

- отображение страниц;
- формы регистрации, входа, поиска и управления;
- загрузку изображений;
- покупку билетов;
- взаимодействие с API.

Backend отвечает за:

- бизнес-логику;
- проверку прав доступа;
- работу с базой данных;
- валидацию данных;
- хранение изображений;
- импорт и экспорт данных.

База данных PostgreSQL хранит основные сущности системы: пользователей, кинотеатры, залы, фильмы, сеансы, билеты и промокоды.

## 5. Основные сущности базы данных

### users

Хранит пользователей системы.

Поля:

- `id` - первичный ключ;
- `username` - имя пользователя, используется как псевдоним;
- `email` - уникальный email для входа;
- `hashed_password` - хеш пароля;
- `role` - роль пользователя: `client` или `admin`;
- `balance` - баланс пользователя.

### cinemas

Хранит кинотеатры.

Поля:

- `id`;
- `name` - уникальное название кинотеатра;
- `address`;
- `halls_count`;
- `description`;
- `image_data` - путь к изображению.

### halls

Хранит залы кинотеатров.

Поля:

- `id`;
- `cinema_id` - ссылка на кинотеатр;
- `name`;
- `total_seats`.

### films

Хранит фильмы.

Поля:

- `id`;
- `title` - уникальное название фильма;
- `director`;
- `operator`;
- `genre`;
- `studio`;
- `actors`;
- `description`;
- `year`;
- `duration_minutes`;
- `image_data`.

### sessions

Хранит сеансы.

Поля:

- `id`;
- `film_id` - ссылка на фильм;
- `hall_id` - ссылка на зал;
- `datetime` - дата и время;
- `price` - цена билета;
- `free_seats` - количество свободных мест;
- `status` - статус: `active`, `cancelled`, `finished`.

В интерфейсе статусы отображаются на русском:

| Значение в БД | Отображение |
| --- | --- |
| `active` | Активен |
| `cancelled` | Отменен |
| `finished` | Завершен |

### tickets

Хранит купленные билеты.

Особенность: билет является архивной записью. Он не зависит от удаления фильма, кинотеатра, зала или сеанса.

Поля:

- `id`;
- `user_id`;
- `session_id`;
- `purchased_at`;
- `seat_number`;
- `film_title`;
- `cinema_name`;
- `hall_name`;
- `session_datetime`;
- `price`.

Архивные поля нужны для того, чтобы билет пользователя не изменился и не пропал после удаления связанных данных.

### promo_codes

Хранит промокоды.

Поля:

- `id`;
- `code` - уникальный код;
- `max_uses` - максимальное количество использований;
- `amount` - сумма пополнения;
- `created_at`.

### promo_redemptions

Хранит факты использования промокодов пользователями.

Поля:

- `id`;
- `user_id`;
- `promo_code_id`;
- `redeemed_at`.

Для пары `user_id` и `promo_code_id` задано уникальное ограничение, чтобы один пользователь не мог использовать один и тот же промокод несколько раз.

## 6. Роли пользователей

В системе используются две роли:

| Роль | Назначение |
| --- | --- |
| Клиент | Просмотр данных, покупка билетов, использование промокодов |
| Админ | Управление пользователями, фильмами, кинотеатрами, залами, сеансами, промокодами, импортом и экспортом |

Роль `guest` не используется. Незарегистрированный пользователь может просматривать открытые страницы, но не может покупать билеты и пользоваться личным кабинетом.

## 7. Авторизация и регистрация

Регистрация включает:

- имя пользователя;
- email;
- пароль.

Ограничения:

| Поле | Ограничения |
| --- | --- |
| Имя пользователя | 5-30 символов, только `A-Z`, `a-z`, `0-9` |
| Email | 8-30 символов, английские буквы, цифры и символы `@ . _ -` |
| Пароль | 8-30 символов, только `A-Z`, `a-z`, `0-9` |

Email уникален и используется для входа. Имя пользователя является псевдонимом.

Пароль не хранится в открытом виде. Перед сохранением он хешируется с помощью `bcrypt`.

После успешного входа backend выдает JWT-токен. Frontend сохраняет токен в `localStorage` и отправляет его в заголовке:

```text
Authorization: Bearer <token>
```

## 9. Работа с билетами

Покупка билета выполняется по следующему алгоритму:

1. Пользователь выбирает активный сеанс.
2. Frontend запрашивает свободные и занятые места.
3. Пользователь выбирает место.
4. Backend проверяет:
   - существует ли сеанс;
   - активен ли сеанс;
   - есть ли свободные места;
   - существует ли выбранное место в зале;
   - не занято ли место;
   - достаточно ли денег на балансе.
5. Создается билет.
6. С баланса пользователя списывается стоимость билета.
7. Количество свободных мест уменьшается.

При покупке в билет записываются архивные данные:

- название фильма;
- название кинотеатра;
- название зала;
- дата сеанса;
- цена.

Это позволяет сохранять билеты даже после удаления исходных объектов.

## 10. Удаление связанных данных

При удалении кинотеатра удаляются связанные залы и сеансы.

При удалении зала удаляются связанные сеансы.

При удалении фильма удаляются связанные сеансы.

При удалении сеанса удаляется только сам сеанс.

Билеты при этом не удаляются. Они остаются у пользователя как архив.

Для старых данных предусмотрен файл `archive.py`, который:

- снимает старый внешний ключ `tickets_session_id_fkey`;
- дозаполняет архивные поля билетов перед удалением связанных объектов.

## 11. Промокоды и баланс

Администратор создает промокоды с указанием:

- кода;
- суммы пополнения;
- максимального количества использований.

Пользователь вводит промокод в профиле. Если промокод существует, не исчерпан и не использовался этим пользователем ранее, баланс пользователя увеличивается на сумму промокода.

В админ-панели есть список промокодов со scroll bar и поиском по:

- коду;
- сумме;
- количеству использований.

## 12. Админ-панель

Админ-панель включает:

- импорт JSON;
- экспорт JSON;
- создание и удаление промокодов;
- список пользователей;
- поиск пользователей;
- изменение роли пользователя;
- удаление пользователя.

Поиск пользователей выполняется по:

- имени пользователя;
- email;
- роли.

В выборе роли доступны только:

- Клиент;
- Админ.

## 13. Импорт и экспорт

Экспорт данных формирует JSON со всеми основными таблицами:

- `users`;
- `cinemas`;
- `halls`;
- `films`;
- `sessions`;
- `tickets`;
- `promo_codes`;
- `promo_redemptions`.

Импорт заменяет текущие данные на данные из JSON-файла.

Особенность импорта билетов: если в старом файле нет архивных полей билета, backend пытается восстановить их по связанному сеансу. Если связанного сеанса нет, используются безопасные значения-заглушки.

## 14. Работа с изображениями

Изображения загружаются как обычные файлы через multipart upload. Base64 не используется.

Ограничения:

- форматы: PNG, JPG/JPEG;
- размер: до 10 МБ.

Файлы сохраняются в:

```text
backend/uploads/cinemas/
backend/uploads/films/
```

В базе данных хранится не сам файл, а путь:

```text
/uploads/cinemas/example.png
/uploads/films/example.png
```

Если у кинотеатра или фильма нет картинки, используется заглушка:

```text
backend/uploads/Placeholder/cinemas.png
backend/uploads/Placeholder/films.png
```

Если в базе указан старый путь, но файл физически отсутствует, frontend также автоматически подставляет заглушку.

| Группа | Назначение |
| --- | --- |
| `/auth` | Регистрация, вход, текущий пользователь, пользователи |
| `/cinemas` | Кинотеатры |
| `/halls` | Залы |
| `/films` | Фильмы |

Примеры:

```text
POST /auth/register
POST /auth/login

GET /cinemas
GET /films

GET /tickets/my
POST /promocodes/apply
GET /admin/data/export
POST /admin/data/import
POST /images/upload
```

## 16. Валидация данных

Валидация реализована на двух уровнях:

Frontend:

- HTML-ограничения в формах;
- отображение подсказок;
- проверка формата и размера изображений.

Backend:

- Pydantic-схемы;
- проверки уникальности email, названия фильма, названия кинотеатра;
- проверка прав администратора;
- проверка статуса сеанса;
- проверка свободных мест;
- проверка баланса пользователя;
- проверка повторного использования промокода.

Ошибки backend возвращаются в понятном виде. Frontend преобразует сложные ответы FastAPI, чтобы вместо `[object Object]` показывался нормальный текст ошибки.

## 18. Настройка базы данных

Подключение к базе указано в `backend/database.py`:

```python
DATABASE_URL = "postgresql://postgres:password@localhost:5432/testdb06"
```

Создание базы:

```sql
CREATE DATABASE testdb06;
```

Таблицы создаются автоматически при запуске backend через:

```python
Base.metadata.create_all(bind=engine)
```

Дополнительные изменения схемы выполняются функцией `upgrade_schema()` в `backend/main.py`.

## 19. Очистка базы данных

Если нужно полностью пересоздать схему:

```sql
DROP SCHEMA public CASCADE;
CREATE SCHEMA public;
GRANT ALL ON SCHEMA public TO postgres;
GRANT ALL ON SCHEMA public TO public;
```

После этого нужно перезапустить backend.

Если схема уже актуальная и нужно удалить только данные:

```sql
TRUNCATE TABLE promo_redemptions, tickets, sessions, halls, films, cinemas, promo_codes, users RESTART IDENTITY CASCADE;
```
